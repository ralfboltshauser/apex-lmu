# v0.4.3 — Local Garage and canonical car history

## Shippable outcome

A new Garage destination shows the cars Apex has actually tracked, total
measured kilometres, tracked drives, first/last driven dates and per-track
distance. Team, year, livery and number variants may be grouped under one car
model only when a conservative versioned rule identifies that model. Unknown
raw labels stay explicit rather than being guessed.

This release completes APX-000006 without replacing the immutable lifetime
ledger or mixing imported/replayed telemetry into it.

## Scope boundaries

Included:

- new sidebar Garage route and responsive Garage view;
- local lifetime summary plus per-model and per-track breakdown;
- conservative canonical vehicle resolver with raw-variant disclosure;
- read-only main-process aggregation and narrow preload DTO;
- empty, read-only, future-schema, corrupt/error and unattributed states;
- compact Settings storage/backup controls linking to Garage.

Excluded:

- cloud profile, account, leaderboard or sharing;
- inferred historic data from before Apex began tracking;
- imported recording or replay mileage;
- per-livery artwork or downloaded car imagery;
- editing/merging aliases in the UI;
- track-layout breakdown while the upstream layout remains unavailable;
- rewriting immutable raw vehicle records.

## Infrastructure and persistence

### Data decision: no schema migration for v1

`apex.sqlite3` already contains the required durable facts:

- `vehicle_models` preserves raw name and class;
- `drive_runs` preserves the track and vehicle relationship;
- `distance_chunks` and `distance_accumulators` preserve measured distance by
  drive run;
- `distance_corrections` preserves adjustments but has no track;
- first/last seen and eligible drive counts are already available.

Do not change `schemaVersion` in this release. Add a read model and query only.
Do not use `telemetry.sqlite3`: it is retention-bounded and can contain imported
recordings, so it is not a lifetime source of truth.

The existing `vehicle_aliases` table remains unused in v1. Activating mutable
database aliases would introduce migration, conflict and recovery behavior that
is unnecessary for the initial curated catalog.

## Canonical vehicle identity

### Contract

Create one shared, versioned, pure resolver with inputs:

- normalized raw vehicle name;
- normalized raw class;
- catalog version.

It returns either:

- a stable canonical model ID, display name, manufacturer and class, plus the
  matched rule ID; or
- `unrecognized`, preserving the raw name and class as its own group.

Rules must be exact aliases or tightly bounded patterns backed by an official
roster and a decoded fixture. Do not derive a model by merely deleting team,
year and number tokens. A manufacturer/class pair may map only if the reviewed
catalog proves it is unique for the supported LMU version.

Store the catalog as reviewable local data, not generated remote content. Unit
tests must enumerate every alias. Updating the catalog later is a normal code
release and retroactively improves read-time grouping without rewriting SQLite.

### Aggregation

Add `StatsDatabase.getGarageStats()` or an equivalent dedicated read method.
It should:

1. query raw vehicle totals and measured distance by `drive_runs.track`;
2. resolve each raw vehicle through the catalog;
3. group only equal canonical IDs;
4. retain the raw variant count and labels in the main process;
5. return a bounded renderer DTO without raw database rows or local paths;
6. sort models by distance, then stable display name; sort tracks by distance.

Use “tracked drives”, not “races” or “race sessions”, because session kind is
not authoritative. A drive count is the number of eligible `drive_runs` with
accepted intervals.

Distance corrections cannot be assigned to a track. Model totals include them,
while track rows include measured chunks/accumulators only. Return an explicit
`unattributedDistanceMm` difference when non-zero; never distribute it across
tracks.

## Renderer and IPC implementation

### Main/preload

- Add `apex:get-garage-stats` in Electron main.
- Expose only `getGarageStats()` through preload.
- Define a versioned DTO in renderer ambient types with:
  - status and schema/catalog versions;
  - tracked-since and total distance;
  - canonical model ID/name/manufacturer/class;
  - measured distance, unattributed adjustment, drive count and dates;
  - bounded per-track aggregates;
  - raw variant count, not raw names by default.
- Preserve existing `getLifetimeStats()` during this release so Settings and
  rollback remain compatible.

### Garage view

- Add `garage` to `ViewId`, navigation copy, guide/discovery copy and routing.
- Lead with total tracked distance and tracked-since wording.
- Render one card/row per canonical model with distance, drive count, track
  count and last driven date.
- Expand or navigate to a model detail showing per-track distance and dates.
- Show “N recorded variants grouped” without implying different physics.
- Unknown labels carry an “Unrecognized LMU label” badge and remain separate.
- Empty state explains that tracking begins only after this Apex version's
  lifetime ledger started; it does not imply zero lifetime LMU experience.
- Mark measured Garage content with the existing feedback-redaction boundary.

### Settings

Keep database health, path redaction, backup, tracked-since and total in
Settings → Data & storage. Replace the duplicate full raw-vehicle listing with
an “Open Garage” action once Garage is available.

## Focused automated tests

### Resolver

- every reviewed alias maps to exactly one canonical ID;
- two known team/livery variants of one model merge;
- different models in one class never merge;
- unknown, blank, malformed and future labels remain separate;
- matching is case/whitespace stable but not over-broad;
- output ordering is deterministic.

### SQLite/read model

Extend `electron/stats-database.node-test.cjs` or add a focused Garage suite:

- seed two raw variants of one canonical model across two tracks;
- verify exact total, per-track distance and drive counts;
- seed another model and an unknown label and verify separation;
- verify distance chunks plus live accumulator are counted exactly once;
- verify a correction changes total but becomes unattributed;
- verify replay, self-test, AI and remote frames remain excluded at ingestion;
- verify close/reopen, future schema, corruption handling and backup behavior;
- verify `getStats()` remains backward compatible.

### Renderer

- navigation and route activation;
- loading, ready, empty, unknown-label, error, read-only and future-schema views;
- model expansion and deterministic sorting;
- long track/car names in English and German at 200%;
- feedback redaction covers measured content;
- Settings opens Garage and retains backup behavior.

## Verification and proof

Create a deterministic seed database with a known proof table:

| Input | Expected Garage result |
| --- | --- |
| Two reviewed raw variants, same canonical model | One model, two variants |
| 1,000,000 mm and 2,500,000 mm on Track A | 3.5 km on Track A |
| 4,000,000 mm on Track B | 4.0 km on Track B |
| 500,000 mm correction without run | 8.0 km total, 0.5 km unattributed |
| One unknown raw label | Separate unrecognized model |
| Replay/AI/remote samples | Zero added distance |

The packaged Windows acceptance must:

1. install the predecessor and create known lifetime distance;
2. back up the ledger and record its SHA-256;
3. upgrade to `0.4.3`;
4. confirm total distance is byte-for-byte equivalent at the integer-mm level;
5. open Garage and match every seeded aggregate;
6. record additional official fixture distance and confirm only one increment;
7. replay the repository recording and confirm Garage does not change;
8. restart and verify the same result offline.

The private large recording may be used only as a negative replay-pollution
audit. Its vehicle, track and identity values must not enter committed fixtures
or proof output.

Electron SQLite runtime and predecessor→target data-preservation proof pass.

## Acceptance criteria

- Garage is a first-class sidebar destination.
- Totals exactly reconcile with the existing lifetime ledger.
- Per-track totals are derived from run-linked measured distance only.
- Corrections without a track remain visibly unattributed.
- Reviewed team/livery variants merge without rewriting raw records.
- Unknown raw labels are never guessed or dropped.
- UI says tracked drives and tracked since; it never claims complete LMU history.
- Replay/import/demo/self-test/AI/remote sources cannot change Garage totals.
- Existing backups and Settings health/recovery behavior remain intact.
- No SQLite schema migration occurs in `0.4.3`.

## Post-release verification

- Upgrade a real installation with existing lifetime data.
- Compare pre/post total distance and backup hashes/metadata.
- Inspect at least one known canonical model and one unknown fallback.
- Drive a short official live fixture/LMU session and confirm distance increases
  once, survives restart and appears under the correct track.
- Run a replay afterward and prove the total remains unchanged.
- Switch English/German and test keyboard navigation and 200% scaling.
- Confirm no new network request occurs while using Garage.

Resolve APX-000006 only after the published installed build passes these checks
and the resolution states the tracked-since and canonicalization limitations.

## Rollback and definition of done

Because no migration occurs, rollback removes the Garage route/read model while
leaving the original ledger untouched. The milestone is done only when seeded
and upgraded databases reconcile exactly, unknown identities remain truthful,
the public package is verified, and APX-000006 is resolved with evidence.
