# v0.4.4 — Adaptive live fuel plan

## Shippable outcome

The Strategy view can explicitly adopt the current official live session and
produce a fuel-only plan from measured clean laps, current fuel, tank capacity,
elapsed distance/time and manual service assumptions. The plan updates when new
evidence materially changes it and explains why a stop moved. It does not claim
that Apex detected a race, nor does it model Virtual Energy, tyres, weather,
traffic, driver swaps or pit-menu configuration.

This is the supportable, verifiable scope of APX-000005. It is deliberately
called a “Live fuel plan”, not a complete live race strategy.

## Scope boundaries

Included:

- manual/live mode in Strategy with per-field provenance;
- fixed-lap and timed-session in-race fuel planning;
- current-lap fractional progress and current fuel;
- conservative measured consumption and lap-time evidence;
- event-driven replanning and change explanations;
- current margin, minimum stops, stints and next modeled stop;
- explicit manual reserve, finish rule, pit-lane loss and refuel rate;
- transient recording audit without persistent calibration.

Excluded:

- automatic practice/qualifying/race detection;
- Virtual Energy allocation or battery-as-VE substitution;
- tyre inventory, compound selection or degradation strategy;
- current/future weather strategy;
- opponent traffic, rejoin position and safety-car prediction;
- driver rules/swaps and pit-service concurrency beyond the existing explicit
  fuel-only assumption;
- durable saved plans;
- notifications that command a pit stop without the user viewing the plan.

## Architecture

### 1. One live fuel input model

Introduce a pure `LiveFuelPlanSnapshot` (name may vary, contract may not) that
contains:

- session ID and `sessionKind: unknown` unless authoritative;
- track and raw/canonical car display identity;
- source and control-owner eligibility state;
- current fuel and tank capacity;
- completed laps and current-lap fraction;
- total laps or scheduled duration plus elapsed time;
- accepted clean fuel/lap and lap-time samples;
- sample count, sample range, last accepted lap and model revision;
- exclusion/staleness reason when learning is paused.

Each adopted strategy input must carry provenance:

- `measured-live` for current session facts;
- `measured-profile` for eligible locally persisted historical samples;
- `manual` for reserve, finish rule, pit loss and refuel rate;
- `unavailable` when absent.

Do not create a second consumption algorithm in `StrategyView`. Extract or
reuse the same pure fuel model used by `FuelCalculatorView` so mean,
conservative rate, race projection and sample semantics cannot diverge.

### 2. Separate observation from replanning

Current fuel may change every telemetry frame, but a strategy model should not
be regenerated at bridge frequency.

Increment `modelRevision` only for:

- an accepted clean lap;
- refuel/pit-exit boundary;
- session/distance reset;
- total-lap or scheduled-duration change;
- manual assumption change;
- reconnect that establishes a new coherent session.

The UI may update current fuel/range continuously through a lightweight
selector. Candidate generation runs only when `modelRevision` or a validated
manual assumption changes. This avoids 50 Hz React churn and oscillating calls.

### 3. Fixed-lap solver

- Compute remaining lap equivalents from total laps, completed laps and
  current-lap fraction.
- Feed current fuel, tank, reserve, expected/conservative consumption and
  current race lap into the existing deterministic candidate engine.
- Preserve integer pit/stint boundaries; only the already-started and final
  distance may be fractional where the finish contract requires it.
- Reject infeasible inputs explicitly rather than clamping into a plausible
  answer.

### 4. Timed-session solver

Extend the timed strategy engine with an in-race input:

- scheduled duration and elapsed time;
- current-lap progress;
- measured expected/conservative lap time;
- finish rule (`line-after-zero` or explicitly configured extra lap);
- current fuel and all pit/refuel assumptions.

The solver must couple pit time back into completed race distance. Subtracting
elapsed time and calling the pre-race solver once is not acceptable: a pit can
change whether another line crossing fits before/after zero. Use bounded
exhaustive discrete search or a proven convergent iteration and return a clear
failure if it cannot settle.

### 5. Plan-change explanation

Keep the last accepted plan summary in memory and compare it with the new one.
Show a concise causal diff when material fields change, for example:

- conservative consumption changed after N clean laps;
- next stop moved from lap A to B;
- minimum stop count changed;
- a refuel reset current stint fuel;
- manual reserve or service assumptions changed.

Do not hide small model changes, but only promote a notification/callout when a
stop lap, stop count or reserve safety state changes. The selected candidate and
all cards/timeline/brief values must come from one immutable result.

## UI implementation

- Pass the live snapshot from `App.tsx` to `StrategyView` even while Strategy
  is the active route.
- Add Manual / Live session modes. Do not automatically switch or arm the live
  plan because session kind is unknown.
- If live facts are insufficient, show exactly what is missing and retain the
  last valid manual result separately.
- Display sample count/range, source and staleness beside consumption and pace.
- Keep manual overrides possible and visibly manual.
- Lead with current fuel margin, conservative range, minimum feasible stops and
  next modeled stop/stint.
- Preserve the existing explicit “Not modeled” list.
- Link the measured Live fuel card to Strategy's live mode once the snapshot is
  ready; do not replace unavailable data with demo values.

## Infrastructure and persistence

- No SQLite migration.
- No new cloud or main-process store.
- Existing local fuel profiles remain backward compatible and keep their
  current raw car/track keys in this release.
- Plans and plan history remain in memory. Saving stays unavailable.
- No bridge contract change is required for the fuel-only milestone.
- Add an optional local audit script for a private recording only if it accepts
  an explicit path/environment variable, emits aggregate data, writes nothing
  durable and is not part of ordinary CI.

## Engine invariants and focused tests

Automate at least:

- `stints.length === stopCount + 1`;
- `pitStops.length === stopCount`;
- stint distances sum to projected remaining distance within documented epsilon;
- every conservative stint fits tank capacity and current-stint fuel;
- finish fuel/reserve is finite and non-negative for feasible plans;
- timed search is deterministic and bounded;
- a pit-time change can alter projected timed distance and is recomputed;
- one lower-consumption clean lap can move a stop later only when the
  conservative model actually crosses the boundary;
- refuelling starts a new stint and invalidates the pit lap as a consumption
  sample;
- AI/remote/replay/mixed-control laps never revise the persistent model;
- current fuel continues to update while the model is stale;
- the same snapshot/assumptions always produce byte-equivalent candidates;
- no NaN, Infinity, negative target or capacity overflow for randomized valid
  inputs.

Focused command sequence:

```bash
npx vitest run src/core/fuel-tracker.test.ts src/fuel-profile.test.ts
npx vitest run src/engine/fuel.test.ts src/engine/fuel-plan.test.ts src/engine/strategy.test.ts
npx vitest run src/views/FuelCalculatorView.test.tsx src/views/StrategyView.test.tsx
npm run i18n:check
npm run lint
```

Add property tests covering fixed/timed distance, fractional progress, finish
rules, boundary fuel quantities, very short remaining sessions and multiple pit
stops.

## Integration and recording proof

### Approved fixture/CI

- Drive the approved recording through strict decode and verify replay does not
  mutate local fuel profiles.
- Use deterministic normalized live frames for multi-lap consumption changes,
  pit entry, refuel, pit exit, disconnect and finish.
- Extend the Windows named-mapping fixture if necessary to publish multiple
  local-player laps and a refuel. Keep fixture data visibly synthetic and out of
  production packages.
- Add packaged Electron assertions that Strategy receives live state while its
  route is active and that candidate generation follows model revisions rather
  than raw frame count.

### Private large-recording audit

Run an explicit local audit through the current decoder and verify:

- all 422,467 frames can be consumed without persisting calibration;
- session/refuel/control transitions do not crash or produce non-finite plans;
- AI and remote intervals cannot add samples;
- output contains only aggregate counts and invariant results.

This audit does not prove race kind, VE, tyre inventory or every car class.

### Real LMU acceptance

On a supported current LMU build:

1. enter a drivable session and explicitly select Live session mode;
2. complete clean laps until the model becomes ready;
3. compare displayed current fuel and lap consumption with LMU;
4. deliberately reduce consumption for subsequent clean laps;
5. verify the conservative model and next stop move only at accepted lap
   boundaries and the explanation names the changed evidence;
6. pit/refuel and verify the new stint/current fuel without learning the pit lap;
7. let AI control a lap and verify the model is paused/stale, not relearned;
8. finish/disconnect and verify the plan stops updating coherently.

Manually calculate one fixed-lap and one timed case from the recorded fuel drops
and line crossings. The engine result must match the audit worksheet within the
documented rounding policy.

## Acceptance criteria

- Live mode is explicit and never claims automatic race detection.
- A plan is unavailable until required measured/manual inputs are valid.
- Fixed and timed plans use current progress and current fuel.
- Lower measured consumption can move a later stop, with a causal explanation.
- Candidate cards, timeline, pit rows, targets and brief agree exactly.
- Replanning is event-driven, deterministic and bounded.
- Unsupported VE/tyre/weather/traffic/driver factors cannot affect results.
- Replay/AI/remote evidence cannot alter persistent calibration.
- No plan, identity or raw recording is uploaded or added to SQLite.
- Full repository, Windows bridge and packaged acceptance pass.

## Rollback and definition of done

Rollback removes live Strategy wiring and leaves existing profiles untouched.
Because there is no schema migration or saved-plan format, downgrade is safe.
The milestone is done only when engine invariants, manual worksheet, packaged
Windows and real LMU acceptance agree and APX-000005 is resolved truthfully.
