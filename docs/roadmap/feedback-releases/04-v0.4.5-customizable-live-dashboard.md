# v0.4.5 — Customizable Live dashboard

## Shippable outcome

The Live dashboard has an explicit edit mode in which users can reorder cards,
choose compact or wide presentation, hide non-critical cards and restore the
default. The configuration survives restart, remains responsive and uses the
same DOM and visual order for pointer, keyboard and assistive-technology users.

This completes the layout-customization portion of APX-000003. It deliberately
does not become a free-pixel canvas or duplicate Overlay Studio.

## Scope boundaries

Included:

- reorder the known Live modules;
- compact/wide span where the module supports it;
- show/hide non-critical modules;
- reset to the shipped default;
- pointer drag as a convenience plus complete keyboard controls;
- versioned, validated local preference;
- identical layout semantics for measured and demo Live views;
- responsive and 200% scaling behavior.

Excluded:

- arbitrary x/y coordinates or overlapping cards;
- arbitrary pixel dimensions;
- per-card font-size, colour or CSS editing;
- user-authored widgets;
- syncing layouts between computers;
- storing UI preferences in SQLite;
- changing the separate transparent race overlay configuration;
- hiding connection/source/provenance status needed to interpret data.

## Layout contract

Create a small pure schema similar to:

```ts
type LiveModuleId = 'track-map' | 'fuel' | 'standings' | 'car-state' | 'events'
type LiveModuleSpan = 'compact' | 'wide'

interface LiveDashboardLayoutV1 {
  readonly version: 1
  readonly modules: readonly {
    readonly id: LiveModuleId
    readonly visible: boolean
    readonly span: LiveModuleSpan
  }[]
}
```

The actual module list must match the current measured/dashboard components at
implementation time. It is an allowlist, not arbitrary persisted JSX.

### Parser and migration behavior

- Validate every stored value before render.
- Reject duplicate IDs and invalid span/visibility values.
- Ignore unknown IDs from a future/newer configuration.
- Append newly shipped module IDs in default order.
- Restore defaults for malformed JSON or unsupported schema versions and record
  a local diagnostic without storing the invalid payload in logs.
- Keep the storage key versioned, for example `apex:live-layout:v1`.
- Export/import is not part of this release.

## Rendering architecture

### One module descriptor list

Refactor measured and demo Live dashboards to render from one ordered descriptor
list. Each descriptor supplies:

- stable module ID;
- measured and demo renderer;
- default span and visibility;
- whether hiding/widening is allowed;
- localized title used by edit controls.

Do not maintain separate hard-coded card order in the measured and demo JSX.
The user preference selects presentation; it never changes which data source a
card receives.

### Grid behavior

- Use normal CSS Grid auto-placement in array/DOM order.
- Implement spans with classes/data attributes, not persisted column numbers.
- Define large, medium and single-column breakpoints.
- At single-column widths, both spans occupy one column without changing the
  saved preference.
- Avoid masonry/absolute positioning and JavaScript measurements.
- Let content determine row height so the `0.4.2` typography scale remains valid.

### Edit mode

- A clearly labelled “Edit layout” button unlocks controls; “Done” exits.
- Each card gets a visible drag handle only in edit mode.
- Pointer drag reorders the underlying array, causing DOM order to change.
- Every module also has keyboard-operable Move before, Move after,
  Compact/Wide and Hide actions.
- Focus follows the moved card and an `aria-live` status announces the new
  position.
- A hidden-modules tray allows restoration.
- Reset asks for confirmation only if the layout differs from default, then
  restores the shipped layout atomically.
- Connection/source header and edit/reset controls are outside the customizable
  module list and can never be hidden.

Save valid changes immediately after each completed action. Do not persist every
pointer-move event during a drag.

## Infrastructure and persistence

- No SQLite migration.
- No Electron main/preload IPC.
- One small localStorage preference owned by a pure parser/serializer.
- No new runtime dependency. Use platform pointer/drag events plus explicit
  keyboard controls; if native drag behavior proves unreliable, implement a
  bounded pointer reorder helper rather than importing a dashboard framework.
- Existing overlay configuration and fuel profiles remain untouched.

## Focused automated tests

### Schema/parser

- default layout is valid and contains every module exactly once;
- round-trip preserves order/visibility/span;
- malformed JSON returns default;
- duplicates, missing fields and invalid spans are rejected safely;
- unknown IDs are ignored and new defaults appended;
- future version cannot crash or erase a usable default;
- reset removes/replaces only the Live-layout key.

### Renderer interactions

- enter/exit edit mode;
- reorder with pointer/drag helper;
- reorder using keyboard only and verify DOM order;
- compact/wide and responsive breakpoint behavior;
- hide and restore each allowed module;
- reset and restart persistence;
- source header cannot be hidden;
- measured and demo layouts use the same order but keep distinct provenance;
- focus and live-region announcement after movement;
- German labels, long card content and 200% scaling;
- feedback selection/redaction continues to cover measured cards.

Focused command sequence:

```bash
npx vitest run src/live-layout.test.ts src/views/LiveView.test.tsx
npm run i18n:check
npm run lint
npm run build
```

Add a Playwright/Electron interaction case for real pointer reorder because DOM
unit events alone do not prove Chromium drag behavior.

## Visual verification matrix

Test default and customized layouts at:

- 1280×720, 1600×1000, 1920×1080 and an ultrawide viewport;
- 100%, 125%, 150% and 200% scaling;
- English and German;
- offline, scoring-only, measured-ready and demo states;
- reduced motion and keyboard-only use.

Verify:

- DOM/focus order matches the visible order;
- cards never overlap or leave unreachable controls;
- hidden cards remain recoverable;
- compact cards do not truncate required units/provenance;
- wide cards use space without imposing fixed pixel widths;
- a single-column viewport remains usable without rewriting saved preferences;
- edit controls disappear completely outside edit mode;
- no layout choice causes a measured card to display demo content.

## Packaged and update proof

1. In `0.4.4`, establish existing lifetime data, fuel profiles and overlay
   configuration.
2. Upgrade to packaged `0.4.5` and confirm those stores are unchanged.
3. Create a custom Live layout, close the app normally and verify it after
   restart.
4. Force-close after a completed reorder and verify the last completed action,
   not a half-drag state, is restored.
5. Corrupt only the Live layout preference and verify safe default recovery plus
   a bounded local diagnostic.
6. Run measured Windows fixture/replay and confirm source/provenance and gradual
   scoring→vehicle states survive reordering/hiding.
7. Confirm Overlay Studio and the transparent HUD retain their independent
   configuration.

## Required full release gate

After schema, renderer, pointer/keyboard, visual and packaged-update checks pass,
run from a clean checkout of the release commit:

```bash
npm ci
npm run i18n:check
npm run lint
npm run build
npm run build:site
npm run test:all
npm run build:bridge:win
npm audit --audit-level=high
```

The mandatory pre-push hook must create and validate the Windows installer,
portable ZIP, `latest.yml` and `SHA256SUMS.txt`. Source-mode drag behavior is
not sufficient; the packaged Chromium pointer test must pass.

## Acceptance criteria

- Every shipped Live module can be reordered without arbitrary coordinates.
- Supported cards can switch compact/wide and non-critical cards can hide.
- Hidden modules and defaults are always recoverable.
- Pointer and keyboard operations produce the same persisted DOM order.
- Layout is responsive and usable at 200% in English and German.
- Malformed/future preferences fail safely without affecting other settings.
- Measured/demo provenance cannot be crossed by the layout abstraction.
- No database, bridge, IPC, network or overlay-config change occurs.
- Full repository and packaged Windows gates pass.

## Post-release verification

- Upgrade a real `0.4.4` installation.
- Reorder every module once with pointer and once with keyboard.
- Change spans, hide/restore modules, restart, reset and restart again.
- Exercise measured and demo Live modes without mixing provenance.
- Repeat at 100% and 200% in English and German.
- Verify existing Garage totals, live fuel profiles and transparent overlay
  configuration are unchanged.
- Confirm the offline in-app release note describes the feature accurately.

Resolve APX-000003 only after this published-build acceptance and reference the
earlier typography release plus this layout release in its resolution summary.

## Rollback and definition of done

Rollback ignores the versioned Live-layout key and falls back to the old fixed
layout; it does not touch durable measured data. The milestone is done when
keyboard/pointer order, persistence, responsive scaling and provenance all pass
in the installed release and APX-000003 is resolved with evidence.
